Бизнес требования

 Необходимо разработать C++ библиотеку UI компонентов для использования в quant проектах с целью визуализации различных бизнес-сущностей (поверхностей, кривых и тд)

  • 2D charts
  • 3D surface
  • Pie charts
  • Histogram
  • Heat map
  • Grid table
  • Hierarchy tree

Спецификация

Применение библиотеки 

  • Библиотека представляет собой динамически линкуемый C++ компонент, используемый в целевом приложении
  • Вызов библиотечных функций происходит непосредственно из кода приложения
  • Все вызовы являются неблокирующими по отношению к хост-приложению
  • Методы библиотеки оперируют бизнес-доменными объектами, таким образом явной сериализации данных перед вызовом не происходит

​Пр​имер использования

 

Вывод можно настраивать, используя специальный объект настроек. Дефолтный конструктор которого заполняет поля пока захардкоженными в код строками. В будущем эти строки планируется брать из файла с конфигурацией. Ниже пример построения поверхности для term_structure_data:

toolkit::toolkit_prefs_tsd_surface new_prefs; - дефолтный конструктор заполняет структуру дефолтными настройками (например, из файла)

new_prefs.chart_title = "TSD surface"; - какие-то из дефолтных настроек мы решаем изменить

new_prefs.y_axix_name = "New Y-axis name";

new_prefs.z_axix_name = L2UTF("Новая Z-ось"); - оси - UTF8 строки. Для использование не ASCII строк следует использовать макрос L2UTF

toolkit.draw_surface(tsd, new_prefs);

 

Благодаря опциональности аргумента с настройками можно осуществить построение графика одной строкой:

toolkit.draw_surface(tsd); - тогда настройки отображения возьмутся дефолтные

 

Хочется отметить, что для построения графиков структур бизнес-логики (например, term_structure_data) мы требуем опциональный аргумент с настройками отображения. А в случае вывода "чистых" точек (не структуры бизнес-логики) мы настройки отображения передаём внутри структуры с точками (например, surface). Кстати, построение графиков для объектов бизнес-логики внутри само использует метод с выводом "чистых" точек. И уже этот ОДИН объект сериализуется и передаётся рендеринг-процессу. Именно поэтому здесь нет второго аргумента, а настройки отображения внутри. Для сериализации!

Дизайн 

1.     Внутреннее устройство библиотеки

Toolkit – отдельный исполняемый файл. Сделано это для того, чтобы: 

    • избежать потери нарисованных графиков в случае падения, из-за ошибки VaR Calculator’а
    • избежать ограничений в объёме пользовательского адресного пространства, которое имело бы место, если бы Toolkit работал в процессе VaR Calculator’а. (Напомню, что на 32-битной машине с Windows это 2GB (или 3GB, если активирован Physical Address Extension)) При текущей архитектуре мы сможем запускать одновременно несколько процессов Toolkit.
    • ​Автономный запуск Toolkit’а для работы с прежде сохранёнными графиками без необходимости загружать сам VaR Calculator.

2.     Описание компонентов библиотеки

В общем случае система будет представлять из себя два исполняемых файла (VaR Calculator и Toolkit) и набор графических библиотек для рисования графиков.

Рассматриваются два варианта построения библиотеки: использующей C++ graphics library и .NET graphics library. В случае использование .NET библиотеки (например, DevExpress) будет необходимо реализовывать промежуточную C++/CLI библиотеку для возможности вызова методов и .NET библиотеки (показано на второй схеме).

Может возникнуть необходимость использовать несколько графических библиотек (например, DevExpress не умеет рисовать 3D Charts) За совмещение работы с ними будет отвечать Toolkit. (Опять же отмечу, что VaR Calculator продолжит работать всё с тем же API, который предоставляет прилинкованная к нему библиотека)

Компоненты системы: 

    1. Хост-процесс – например, VaR Calculator
    2. Клиентский API – библиотека (.lib), которая будет линковаться к исполняемому файлу VaR Calculator’а и предоставлять интерфейсы для работы с Toolkit. Это сделано для того, чтобы избежать изменений в коде VaR Calculator’а в случае изменений graphics library или IPC (например, socket вместо pipe, если захотим Toolkit запускать на другой машине в сети)
    3. Interop library – код для связи хост-процесса и рендер-процесса (IPC library). Сейчас планируется, что это будет named pipe. Но, если захочется использовать другой IPC (например, socket), то сделать такое изменение будет не сложно. ​Объекты  клиентского API’а, которые мы захотим отдавать в Toolkit силами IPC будут сериализоваться библиотекой.
    4. Рендерер-процесс – второе приложение, которое будет отвечать за работу с graphics library и предоставляющее дополнительную функциональность (например, «многостраничность» приложения, сохранение/загрузка графиков). Это приложение будет общаться с Toolkit library VaR Calculator’а посредством библиотеки IPC.
    5. Graphics library – сторонняя библиотека, которую мы будем использовать для построения графиков.

В случае использования .NET библиотеки (например, DevExpress) для связи C++ кода с библиотекой будет необходимо использовать ещё один компонент:​​

 

6. C++/CLI .dll file – библиотека для вызова .NET процедур из C++ кода. C++/CLI – модификация C++ (раньше это был Managed C++). Можно рассматривать его как новый язык программирования, изучение работы с которым требует дополнительного времени.
